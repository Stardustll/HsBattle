# HB 当前回合多步规划设计

## 背景

当前 `HB框架策略(实验)` 已具备独立的快照适配、留牌决策、战斗决策入口，但战斗决策仍是“基于当前 options 的单步打分”，本质上属于局部贪心选择：

- 能给单个动作打分；
- 能为单个动作选目标；
- 不能把 `出牌 + 英雄技能 + 攻击` 放进同一条序列统一规划；
- 不能显式比较“先解再铺”和“先铺再解”这类顺序差异。

本次需求已经收敛为：

1. 只做 **当前回合内** 的多步规划，不做跨回合规划；
2. 把 **出牌 + 英雄技能 + 攻击** 纳入 **同一条动作序列** 统一评估；
3. 决策倾向为 **控场优先**：
   - 先解关键随从；
   - 场面安全高于盲目打脸；
   - 但若存在本回合斩杀，仍优先斩杀；
4. 不接受靠大量近似去“猜”复杂结算；
5. 第一版采用 **精确但限范围** 的方案：支持范围内严格推演，超出范围则停止该分支或回退。

## 目标

为 `HB框架策略(实验)` 增加一套 **当前回合多步规划器**，使其能够：

1. 在当前回合内搜索多步动作序列；
2. 在支持范围内对动作后的状态做精确推演；
3. 统一比较不同出牌顺序、技能顺序、攻击顺序的终局优劣；
4. 返回“最佳序列中的第一步”给现有 `BattleController` 执行；
5. 在遇到复杂、无法精确模拟的动作时安全停止该分支，并在必要时回退到现有 HB 单步决策或 legacy。

## 非目标

本阶段明确不做：

- 不实现完整 Hearthstone 规则引擎；
- 不做 2~3 回合中期规划；
- 不做卡组专属、职业专属脚本；
- 不把所有复杂战吼、亡语、随机效果都纳入模拟；
- 不一次返回整条序列给控制器无脑连点执行；
- 不牺牲稳定性去追求“全牌池全精确”。

## 方案选型

### 方案 A：全量精确规则模拟器

优点：

- 理论上最接近真正的“整回合统一规划”；
- 可以把所有动作都放进同一棵搜索树。

缺点：

- 实际上等价于在项目内重建一套小型炉石规则引擎；
- 复杂效果、连锁结算、随机、光环、改费、衍生选项会使范围失控；
- 不符合本阶段可交付边界。

### 方案 B：精确但限范围的回合模拟器（推荐）

优点：

- 支持范围内可做真正的精确推演；
- 能统一规划 `出牌 + 英雄技能 + 攻击`；
- 工程体量可控，便于逐步扩展；
- 可与现有 `HB`/`legacy` 路由自然衔接。

缺点：

- 第一版必须明确“支持什么、不支持什么”；
- 某些复杂局面仍会回退。

### 方案 C：每步真实执行后重算，但不做整回合搜索

优点：

- 与真实游戏状态最贴近；
- 实现最稳。

缺点：

- 本质上仍是增强版单步决策；
- 不能提前比较整条序列的顺序优劣；
- 不满足本次“统一规划整条动作序列”的核心目标。

## 设计结论

采用 **方案 B：精确但限范围的回合模拟器**。

具体来说：

- 在支持矩阵内，规划器使用本地可演化状态做 **精确状态转移**；
- 在搜索阶段统一评估完整动作序列的终局质量；
- 执行阶段只取 **最佳序列的第一步**，每执行一步都重新从真实 `GameState` 抓取状态并重新规划；
- 遇到不支持的复杂动作时，不做近似猜测，而是停止对应分支扩展；必要时回退。

## 总体架构

### 1. 快照层

扩展 `HbSnapshotAdapter`，让战斗快照不再只包含“当前 option 基础信息”，而是补足当前回合规划所需的事实层：

- 当前/剩余法力；
- 手牌实体、费用、类型、是否需要目标；
- 双方英雄状态；
- 双方场上角色状态；
- 英雄技能可用性；
- 可攻击单位与攻击次数状态；
- 当前合法动作及目标集合。

快照层只负责提取 **当前真实事实**，不负责推演未来状态。

### 2. 可演化状态层

新增 `HbSimulatedTurnState`，用于搜索树中的节点状态。该状态不再直接依赖 Unity/游戏实体对象，而是保存规划器可以安全修改的纯数据：

- 剩余法力；
- 手牌区实体摘要；
- 己方/敌方场面角色摘要；
- 英雄状态；
- 英雄技能是否已使用；
- 哪些角色已经攻击过；
- 当前已执行的动作序列；
- 当前局面评分缓存。

### 3. 动作支持判定层

新增 `HbActionSupportClassifier`，对当前合法动作做支持判定：

- `SupportedExact`
- `UnsupportedComplex`

只有被判定为 `SupportedExact` 的动作才允许进入搜索树。

### 4. 动作解析层

新增 `HbActionResolver` 及其细分解析器，对支持范围内的动作执行精确状态转移：

- 普通攻击解析；
- 固定伤害英雄技能解析；
- 固定伤害单体法术解析；
- 普通随从上场解析；
- 结束回合解析。

### 5. 回合规划层

新增 `HbTurnPlanner`，负责：

1. 接收 `HbBattleSnapshot`；
2. 建立初始 `HbSimulatedTurnState`；
3. 枚举当前合法动作；
4. 做受限深度搜索；
5. 返回最佳动作序列；
6. 输出序列第一步作为当前 `StrategyActionPlan`。

### 6. 序列评分层

新增 `HbTurnSequenceScorer`，对搜索节点的终局状态进行 **整序列评分**，而不是继续沿用单步动作打分。

### 7. 引擎集成层

`HbBattleDecisionService` 从“单步打分器”调整为：

1. 构建战斗快照；
2. 调用 `HbTurnPlanner`；
3. 取最佳序列的第一步；
4. 若规划器不可用，则回退到现有 HB 单步决策；
5. 若 HB 单步决策也不可用，则由现有路由回退到 legacy。

## 状态模型

### 1. 角色状态

规划器中的角色状态至少应包含：

- `EntityId`
- 阵营
- 是否英雄 / 随从 / 武器来源
- 攻击力
- 当前生命值
- 最大生命值
- 是否已攻击
- 是否可攻击
- 是否嘲讽
- 是否圣盾
- 是否潜行
- 是否冻结
- 是否受伤

这些字段必须来自当前真实局面，且能支持攻击与伤害结算。

### 2. 手牌状态

每张手牌至少应包含：

- `EntityId`
- 当前费用
- 卡牌类型
- 是否需要目标
- 是否需要站位
- 当前动作支持级别

### 3. 行动状态

当前回合状态至少应包含：

- 总法力
- 剩余法力
- 英雄技能是否可用
- 是否已结束回合
- 当前动作序列
- 当前终局分数

## 支持矩阵

### 第一版纳入精确模拟的动作

第一版只允许下列动作进入搜索树：

1. 普通随从攻击随从；
2. 普通随从攻击英雄；
3. 英雄攻击；
4. 固定伤害英雄技能；
5. 固定伤害单体法术；
6. 无额外结算的普通随从上场；
7. 无额外复杂效果的简单单体动作；
8. `EndTurn`。

### 第一版排除的动作

以下动作一律视为 `UnsupportedComplex`：

- 随机伤害 / 随机召唤；
- 发现、抉择、适应、衍生选项再选择；
- 亡语连锁；
- 抽牌、复制、偷取、变形、额外召唤等复杂战吼；
- 持续光环；
- 临时改费；
- 法强、连击、连锁触发；
- 奥秘、反制、冻结、免疫、复生、磁力等特殊规则；
- 无法从当前快照严格解析结果的任何动作。

### 支持范围的原则

本设计中的“精确”含义是：

> 进入搜索树的动作，其状态转移结果必须由当前状态和动作本身严格推出，而不是通过猜测或近似估计。

若无法满足这一点，则该动作不应进入搜索。

## 搜索设计

### 1. 搜索策略

采用 **受限深度的 best-first / beam search**，而不是全量穷举。

原因：

- 当前回合的动作组合会随目标、费用、攻击顺序快速膨胀；
- 需要优先探索更有希望的控场序列；
- 需要明确的复杂度上限。

### 2. 节点定义

搜索树中的每个节点表示一次中间局面：

- 一个 `HbSimulatedTurnState`
- 一条已执行动作序列
- 当前局面分数

### 3. 扩展顺序

为贴合“控场优先”，动作扩展顺序优先考虑：

1. 能击杀敌方关键随从的攻击 / 法术 / 技能；
2. 能保住我方攻击者存活的交换；
3. 能补充场面的普通上场；
4. 打脸动作；
5. `EndTurn`。

扩展顺序不是最终评分，只是让搜索更快触达高质量控场序列。

### 4. 深度与剪枝

第一版建议：

- 最大动作深度：`6~8`；
- 同状态去重：手牌、剩余法力、场面、已攻击标记一致时合并；
- 无收益终止：若只剩低价值动作，则允许提前终止并进入 `EndTurn` 比较；
- 劣分支剪枝：同层明显更差的分支直接丢弃。

### 5. 斩杀覆盖规则

无论默认策略倾向如何，只要存在支持范围内的 **本回合必杀序列**，该序列直接压过普通控场分。

## 评分设计

### 1. 评分目标

评分对象是“执行到当前节点后的局面”，不是单个动作本身。

### 2. 核心评分项

整序列评分由以下项构成：

- `enemy_board_threat_penalty`
  - 敌方存活场面的威胁罚分；
- `friendly_board_value`
  - 我方存活场面的价值加分；
- `clean_trade_bonus`
  - 我方不死且解掉敌方关键单位的奖励；
- `bad_trade_penalty`
  - 我方送掉高价值单位却未换到足够收益的罚分；
- `mana_usage_bonus`
  - 费用利用率加分；
- `face_damage_bonus`
  - 打脸伤害加分；
- `empty_end_turn_penalty`
  - 明显还有收益动作时过早结束回合的罚分。

### 3. 控场优先落地

控场优先并不意味着永远不打脸，而是：

- 在敌方场面威胁高时，解场分显著高于打脸分；
- 在敌方场面已被压低、或者接近斩杀时，打脸权重上升；
- 交换后我方能保住关键随从时，给予高额奖励；
- “清掉关键威胁后再铺场”通常应优于“先贪铺再交换”。

### 4. 关键威胁定义

第一版不做卡牌语义特化，因此关键威胁主要依赖通用属性：

- 敌方攻击力高；
- 敌方血量使其难被顺手处理；
- 该单位会造成高额返伤或阻断我方节奏；
- 该单位带有嘲讽等会影响回合序列的通用关键词。

## 执行与回退策略

### 1. 不整包执行序列

即便规划器能输出一整条最佳序列，控制层也不应一次性缓存整包动作并全部执行。

更稳的方式是：

1. 搜索得到最佳序列；
2. 只返回第一步 `StrategyActionPlan`；
3. 执行该动作；
4. 下一 Tick 重新从真实 `GameState` 建快照；
5. 重新规划下一步。

这样可以在不牺牲整回合搜索收益的前提下，始终对齐真实游戏状态。

### 2. 分支级回退

当某个动作被判定为 `UnsupportedComplex` 时：

- 只丢弃该动作分支；
- 继续搜索其他可精确解析的动作。

### 3. 回合级回退

当当前局面中：

- 高价值动作大多不可精确解析；
- 或规划器无法形成可靠序列；
- 或模拟器发生异常；

则本回合回退到：

1. 现有 HB 单步决策；
2. 必要时再由既有路由回退到 legacy。

### 4. 回退要求

回退必须满足：

- 不阻塞当前对局；
- 不返回不合法动作；
- 不刷屏日志；
- 不改变现有 `BattleController` 对动作执行的入口契约。

## 与现有系统的集成方式

### 1. `HbSnapshotAdapter`

需要从“当前 option 基础快照”升级为“可用于当前回合规划的事实快照”。

### 2. `HbBattleDecisionService`

职责调整为：

- 调用分类器过滤动作；
- 调用规划器搜索最佳序列；
- 将序列第一步映射回 `StrategyActionPlan`；
- 失败时回退到现有 HB 单步逻辑。

### 3. `HbStrategyEngine`

继续维持当前统一接口，不扩散变更范围。对 `BattleController` 来说，HB 仍然只是返回一个当前要执行的 `StrategyActionPlan`。

### 4. `BattleController`

无需改成“执行动作列表”。现有控制器只需继续按当前机制执行单步动作，并在下一次 Tick 再次请求策略结果。

## 测试方案

### 1. 分类器测试

验证：

- 简单攻击 / 固定伤害动作会被识别为 `SupportedExact`；
- 复杂战吼 / 随机 / 发现类动作会被识别为 `UnsupportedComplex`。

### 2. 解析器测试

验证：

- 攻击交换后的双方血量、死亡、圣盾变化正确；
- 固定伤害技能 / 法术会精确扣血与移除目标；
- 普通随从上场会正确扣费并改变场面；
- 英雄技能使用状态正确更新。

### 3. 规划器场景测试

第一批至少覆盖以下场景：

1. 两种解场顺序中，只有一种能形成优质交换；
2. `先攻击再出牌` 优于 `先出牌再攻击`；
3. `先英雄技能补刀再随从交换`；
4. 费用有限时，优先解掉高威胁敌方随从而不是贪满费；
5. 存在支持范围内斩杀时直接走斩杀；
6. 存在复杂动作时，该动作分支被剪掉但搜索仍可继续；
7. 所有高价值动作都复杂时，稳定回退。

### 4. 回归测试

保留并扩展现有 `tests/*.ps1` 回归脚本，验证：

- 规划器类、分类器、解析器和接线调用存在；
- HB 路由未破坏 legacy 回退；
- 现有策略配置和 UI 回归继续通过；
- Release 构建通过。

## 落地步骤

建议按以下顺序实施：

1. 扩充战斗快照结构；
2. 新增 `HbSimulatedTurnState`；
3. 新增动作支持分类器；
4. 先实现攻击和固定伤害动作解析；
5. 再实现普通上场与英雄技能解析；
6. 新增序列评分器与回合规划器；
7. 改造 `HbBattleDecisionService`，返回最佳序列第一步；
8. 补纯策略测试与回归脚本；
9. 最后再做权重调优。

## 风险与控制

### 风险 1：支持范围过窄，收益不明显

控制方式：

- 优先覆盖最常见、最能体现收益的动作：
  - 攻击
  - 英雄技能
  - 固定伤害
  - 普通随从上场
- 让“控场优先”和“顺序优化”先可见。

### 风险 2：搜索复杂度失控

控制方式：

- 使用 beam search；
- 限制最大深度；
- 做状态去重和分支剪枝；
- 禁止复杂动作进入搜索树。

### 风险 3：规划结果与真实状态漂移

控制方式：

- 只执行最佳序列的第一步；
- 每步后重新从真实 `GameState` 抓快照；
- 不缓存长序列直接执行。

### 风险 4：回退路径失效

控制方式：

- 维持 `HB -> HB 单步 -> legacy` 的清晰回退层级；
- 为回退场景增加专项测试。

## 最终设计结论

本次不做完整规则引擎，而是在 `HB框架策略(实验)` 中新增一套 **支持范围内精确、复杂动作不入搜索、执行时逐步重对齐真实状态** 的当前回合多步规划器。

它的核心收益不是“全牌池全精确”，而是：

- 能把 `出牌 + 英雄技能 + 攻击` 放进同一条序列统一比较；
- 能稳定选出“先解后铺、先补刀后交换、先处理威胁再发展场面”的第一步；
- 能在复杂局面下安全停止搜索并回退，而不是用近似策略硬猜。
